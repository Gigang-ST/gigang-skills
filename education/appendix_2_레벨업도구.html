기강 · Claude Code 한 걸음 더

에이전트를 잘 부리는 도구 4종

기본기를 익혔다면 이제 “더 똑똑하게 일 시키는” 방법입니다. 끝을 정의하고(/goal), 짓기 전에 설계하고(brainstorming), 웹을 직접 다루고(browse), 깊이 조사해 정리하는(deep-research) 네 가지를 소개합니다.

왜 이 셋인가요?

처음엔 “이거 해줘 → 답 받기”의 한 번씩 주고받기로 충분합니다. 하지만 에이전트(스스로 여러 단계를 밟는 AI)를 제대로 쓰려면, 일을 맡기는 방식 자체를 한 단계 올려야 합니다. 아래 셋은 그 핵심을 각각 담당합니다.

① /goal끝나는 조건을 걸고 자율 실행
② brainstorming만들기 전에 설계부터
③ gstack의 웹·문서 도구browse로 웹을 직접 다루고 PDF까지
④ /deep-research여러 출처를 교차검증해 보고서로
1

/goal — 끝나는 조건을 정해주고 맡기기

한 번 답하고 멈추는 대신, “이 조건이 될 때까지” 알아서 계속 일하게 하는 명령
자율 실행 (Autonomous)
한 줄 비유
“다 되면 알려줘” 하고 자리를 비우는 것. 끝나는 지점(결승선)만 정해주면 AI가 스스로 거기까지 달려갑니다.
어떻게 동작하나
  • 보통 Claude는 한 번 시키면 한 번 답하고 멈춥니다. /goal은 “완료 조건”을 걸어두는 명령입니다.
  • 매 턴이 끝날 때마다, 일하던 AI와는 별개인 작고 빠른 ‘심판’ 모델이 대화를 읽고 “조건 달성됐나?”만 판정합니다. 아니면 한 턴 더, 맞으면 스스로 종료하고 사람에게 넘깁니다.
  • 핵심은 ‘일하는 AI’와 ‘끝났는지 판단하는 AI’를 분리한 점입니다. 자기가 짠 코드라 “다 됐다”고 착각하는 편향을 없애줍니다.
  • 멈추고 싶으면 /goal clear.
잘 쓰는 법 — 조건은 “증명 가능한 것”으로

✕ 나쁜 조건

  • “사용자가 만족할 때까지” — 측정 불가
  • “완벽해질 때까지” — 끝이 없음

✓ 좋은 조건

  • “test/auth 의 모든 테스트 통과”
  • AI가 직접 실행해 결과가 대화에 남는 것

주의: 조건이 모호하면 끝없이 돌며 토큰·시간을 태웁니다. 명확히 측정되는 조건일 때만 쓰고, 켜둔 동안 비용이 쌓인다는 걸 기억하세요.

핵심

AI가 스스로 ‘증명’할 수 있는 결승선을 정해주면, 그 지점까지는 알아서 달린다. 모호한 결승선은 금물.

2

/superpowers:brainstorming — 짓기 전에 설계부터

막연한 아이디어를, 질문을 주고받으며 “무엇을·왜·어떻게”가 분명한 설계로 다듬는 과정
설계 우선 (Design-first)
한 줄 비유
목수에게 “책상 만들어줘”라고 바로 말하는 대신, 용도·크기·예산을 함께 정하고 도면을 그린 뒤 톱질을 시작하는 것.
어떻게 동작하나
  • 질문을 한 번에 하나씩 던지며 목적·제약·성공 기준을 함께 명확히 합니다. (한꺼번에 쏟아붓지 않고, 가능하면 객관식으로)
  • 그다음 2~3가지 접근법을 장단점과 함께 제안하고, 합의된 설계를 문서로 남깁니다.
  • 중요한 규칙(HARD GATE): 설계가 승인되기 전에는 코드를 단 한 줄도 짜지 않습니다.
왜 처음 쓰는 사람에게 특히 좋은가
  • “그냥 만들어줘”라고 막연히 던지면, AI는 엉뚱한 걸 빠르게 잘 만들어버립니다. 헛수고가 가장 큰 낭비예요.
  • 먼저 생각을 같이 정리하면, “내가 진짜 원한 게 이거였구나”를 코드를 짜기 전에 발견합니다.
  • “이건 너무 단순해서 설계 필요 없어”가 바로 함정입니다 — 단순해 보이는 일일수록 가정이 어긋나 있기 쉽습니다.

언제 쓰나: 새 기능·도구·자동화 등 뭔가를 만들기 시작할 때. 요청이 한 문장으로 막연하게 느껴질 때 특히.

핵심

만들기 전에 5분 대화로 설계를 맞추면, 잘못 만든 걸 갈아엎는 5시간을 아낀다.

3

gstack의 웹·문서 도구 — AI가 웹을 “직접 조작”하게

gstack은 원래 개발자용 도구 모음이지만, 사무직에게 진짜 쓸모있는 건 함께 딸려오는 웹·문서 자동화 묶음입니다
웹·문서 자동화 (Office)
한 줄 비유
검색 엔진에 물어보는 것(읽기)과, 누군가 대신 브라우저를 켜고 클릭·입력해주는 것(조작)의 차이. gstack의 browse는 후자입니다 — 한 번 깔아두면 웹 작업이 필요할 때 Claude가 알아서 꺼내 씁니다.
왜 검색·요약(WebSearch / WebFetch)보다 좋은가
  • WebSearch는 검색 결과를, WebFetch는 한 페이지 텍스트를 ‘읽어’오기만 합니다.
  • browse는 진짜 브라우저를 열어 버튼 클릭, 폼 입력, 로그인된 페이지 읽기, 화면 캡처, 표·목록 추출까지 ‘조작’합니다. — 즉 눌러봐야 나오는 내용도 가져옵니다.
사무직에게 쓸모있는 gstack 도구
  • /browse — 보이지 않게(헤드리스) 빠르게 페이지를 열어 캡처·추출·양식 채우기. 명령당 약 0.1초.
  • /connect-chrome — 보이는 진짜 크롬을 띄워 AI가 하는 행동을 눈으로 보며 진행. 로그인이 필요하면 내가 직접 로그인하고 그다음을 AI에 넘기는, 안전한 방식에 딱 맞습니다. (부록 1의 “로그인은 사람이” 원칙과 연결됩니다.)
  • /setup-browser-cookies — 내 브라우저의 로그인 세션(쿠키)을 골라 가져와, 인증이 필요한 사내 페이지도 다룰 수 있게.
  • /make-pdf — 마크다운 문서를 여백·페이지번호·표지·목차가 갖춰진 깔끔한 PDF로 변환. 보고서 마무리용.

예: “사내 대시보드 이 숫자 캡처해줘”, “이 신청 양식 채워줘”, “경쟁사 가격 페이지 표로 정리해줘”, “이 회의록 PDF로 만들어줘”.

참고: gstack에는 코드 리뷰·QA·배포 같은 개발자용 명령이 더 많습니다(/review, /qa, /cso, /ship 등). 사무직이라면 위 웹·문서 묶음만 챙기면 충분합니다. (설치 필요 · YC 대표 Garry Tan이 공개한 오픈소스)

핵심

‘읽기’만 하는 검색을 넘어, AI가 실제 브라우저를 대신 조작하게 하면 사무 업무의 웹 작업이 자동화된다. 로그인은 /connect-chrome로 사람이 직접.

4

/deep-research — 깊게 조사해 “출처 달린 보고서”로

한 가지 질문을 여러 갈래로 검색·교차검증해서, 근거가 붙은 정리본을 만들어주는 조사 도구
심층 조사 (Research)
한 줄 비유
한 명에게 “대충 찾아봐”가 아니라, 여러 조사원을 동시에 풀어 자료를 모으고, 서로 말이 맞는지 대조한 뒤, 한 편의 출처 달린 보고서로 묶어오는 것.
어떻게 동작하나
  • 질문을 여러 검색으로 동시에 펼쳐 자료를 폭넓게 모읍니다.
  • 모은 출처를 읽고, 주장끼리 교차로 따져(검증) 틀리거나 근거 약한 내용을 걸러냅니다.
  • 마지막에 출처(링크)가 붙은 하나의 보고서로 정리해줍니다 — 어디서 나온 말인지 추적 가능.
언제 쓰나 (사무직 활용)
  • 시장·경쟁사 조사, 업계 동향 브리프, 배경 자료 정리, “이 주제 조사해서 정리해줘” 류.
  • 단순 검색 한두 번으로 끝낼 일에는 과합니다. 여러 출처를 비교·검증해야 하는 묵직한 조사에 어울립니다.

잘 쓰는 법: 질문을 구체적으로 주세요. 예: “전기차 보조금” ✕ → “2026년 서울시 전기 승용차 보조금 조건과 금액” ○. 범위가 막연하면 도구가 먼저 2~3가지를 되물어 좁힙니다.

핵심

검증 안 된 검색 요약 대신, 여러 출처를 교차검증한 ‘출처 달린 보고서’가 필요할 때 쓴다. 질문은 구체적으로.
