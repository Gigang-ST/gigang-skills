기강 · Claude Code 입문 가이드

처음 쓸 때 흔히 하는 실수 & 꼭 알아야 할 것

오늘 Claude Code를 처음 만져보는 분들을 위해, 자주 밟는 지뢰들만 추렸습니다. 모임원들이 직접 겪은 사례를 사실확인해 정리했어요.

이 문서는 왜 있나요?

Claude Code는 강력하지만, 처음 쓰면 돈이 새거나(과금), 보안에 걸리거나, 결과물이 어설프거나 하는 실수를 하기 쉽습니다. 아래 5가지만 미리 알아두면 대부분 피할 수 있습니다.

기여 · 김석준김혜빈박송이양현모
1

“API 키를 넣어라”는 안내를 만났을 때

제보 · 김석준 · 김혜빈
✓ 사실입니다
어떤 상황이냐면
대부분 일부러 그러는 게 아닙니다. 작업을 시키다 보면 Claude가 “Anthropic API 키를 발급받아 이렇게 호출하면 된다”고 친절히 안내하고, 처음 쓰는 사람은 그게 정답인 줄 알고 그대로 따라갑니다. 그런데 그 길은 구독과 별개로 돈이 따로 나가는 종량제 경로예요.
사실확인 & 보충

과금 구조는 실제로 분리돼 있습니다.

  • 구독(Pro/Max) — 매달 정해진 한도 안에서 쓰는 정액제. 한도는 주기적으로 리셋.
  • API 키 — 쓴 토큰만큼 차감하는 종량제. 콘솔에 결제수단/크레딧을 따로 등록해야 하고, 헤비하게 쓰면 구독보다 쉽게 비싸집니다.

그러니 Claude가 “API 키로 이렇게 하면 됩니다”라고 제안하더라도 곧바로 받아들이지 말고, “API 없이, 너(Claude Code)가 직접 처리하는 방식으로 다시 짜줘”라고 되돌려 시키세요. 대부분의 개발·학습 작업은 그렇게 구독 안에서 끝낼 수 있습니다.

이렇게 돌려서 개발시키세요

✕ 무심코 따라가기

  • “API 키 발급받아 넣으세요” → 그대로 발급·결제
  • 도우미가 할 일을 종량제 API로 중복 결제

✓ 되돌려 시키기

  • “API 말고 네가 직접 해주는 방식으로 바꿔줘”
  • “이거 구독 안에서 가능한 방법으로 다시 설계해줘”

단, 헷갈리기 쉬운 예외가 하나 있습니다 — 만들려는 결과물 자체가 “실행 중에 스스로 AI를 호출하는 앱/자동화”라면, 그 프로그램이 돌아갈 때는 API 키가 불가피합니다(이건 정상). 즉 ‘나를 도와주는 Claude’는 구독으로, ‘내 프로그램이 스스로 부르는 AI’만 API로 구분하면 됩니다.

핵심

Claude가 API 키를 권해도 곧장 따르지 말고 “API 없이 직접 해주는 방식으로 다시 짜줘”라고 되돌려 시킬 것.

2

AI로 메일·문서를 만들 때 (특히 남에게 가는 것)

제보 · 김혜빈
✓ 타당합니다
원래 메모
“AI 요약 메일은 좋지만, 사람들이 ‘AI 냄새’를 맡으면 썩 좋아하지 않더라. 요약·정형화된 양식은 좋지만 마지막에 사람이 검수하는 역할은 빼지 말 것. 특히 결과물이 남에게 가는 문서일 때. 사람 전달용 문서는 손이 더 가고, PPT류는 특히 그렇다.”
사실확인 & 보충
  • “AI 냄새”는 실재합니다. 과한 미사여구, 똑같은 구조, 불릿 남발, 영혼 없는 정중함 — 받는 사람은 금방 알아채고 ‘성의 없다’고 느낍니다.
  • AI는 초안·요약·구조화엔 강하지만, 이 사람·이 맥락에 맞는 톤과 사실 정확성의 최종 책임은 사람 몫입니다.
  • 외부로 나가는 문서일수록 사람 검수(human-in-the-loop)가 필수: AI 초안 → 사람이 사실·톤·민감정보 점검 → 발송.
  • PPT·디자인 산출물은 텍스트/구조까진 AI가 빠르게 뽑지만, 레이아웃 미세조정·브랜드 일관성은 아직 손이 많이 갑니다. “한 방에 완성”을 기대하지 마세요.
핵심

AI에게 80%까지 맡기되, 마지막 20%(검수·톤·고유 정보)는 반드시 사람이 마무리.

3

여러 컴퓨터에서 같은 작업을 이어가고 싶다면

제보 · 박송이
✓ 정확합니다
원래 메모
“멀티 머신으로 코드 작업을 하고 싶으면 GitHub를 써보는 것이 좋을 수도 있다.”
사실확인 & 보충
  • 집·회사·노트북 등 여러 기기에서 같은 작업을 이어가는 표준 방법이 Git + GitHub(원격 저장소)입니다.
  • USB나 메신저로 파일을 주고받으면 곧 버전이 꼬입니다. Git은 변경 이력과 동기화를 정식으로 관리합니다.
  • 흐름은 간단합니다: 머신 A에서 commit → push, 머신 B에서 pull 받아 이어서 작업.
  • 덤으로 백업·되돌리기·협업까지 한 번에 해결됩니다.

⚠ 주의

  • 회사 코드를 public(공개) 저장소에 올리지 않기
  • 비밀번호·API 키를 커밋하지 않기

✓ 권장

  • private 저장소 사용
  • 비밀키는 .gitignore로 제외
핵심

기기 간 작업 = Git/GitHub. 단, 회사 코드·비밀키는 private + 커밋 금지.

4

“일회성 / 주기 / 상시” 부터 구분하라 (토큰 경제)

제보 · 양현모
✓ 타당 · 일부 일반화 필요
원래 메모
“일회성인지 주기성인지 상시인지 구분은 항상 필요하다. 분류 없이 돌리면 — 예: 급등주 추적기인데 매 순간 전체 표와 뉴스를 1분 간격으로 AI에 분석시키면 — 토큰이 남아나지 않는다.”
사실확인 & 보충

전적으로 맞습니다. 자동화를 켜기 전에 “빈도가 곧 비용”이라는 걸 기억하세요.

  • 상시 (예: 급등주 추적기) — 트리거 기반으로. 먼저 싼 코드 규칙으로 조건을 거르고, 조건을 만족할 때만 AI 심층 분석. 매분 전체를 AI에 넣지 마세요.
  • 주기 (예: 주간 매출 분석기) — 단위기간을 명확히 하고, 주기가 도래했을 때 정해진 절차로 분석.
  • 일회성 (예: 현재 상황 가정 → 방향 찾기) — 탐색·의사결정 보조. 대화로 충분하고, 굳이 자동화/API화할 필요 없음.

안티패턴: 급등주 추적기인데 1분마다 전체 종목표 + 뉴스를 통째로 AI에 던지기 → 99%는 “변화 없음”인데 매번 비싼 분석을 돌리니 토큰 폭발.

“차라리 더 싼 GPT 토큰” 부분 보강: 진짜 요점은 GPT냐 Claude냐가 아니라 작업 난이도에 모델 등급(=비용)을 맞추는 것입니다.

  • 고빈도·단순 판별 → 경량·저가 모델(예: Claude Haiku 등)로 1차 필터.
  • 진짜 심층 분석만 → 상위 모델로.
  • 1차 필터를 아예 AI 없이 코드 규칙으로 끝낼 수 있으면 그게 가장 쌉니다.
핵심

켜기 전에 일회성/주기/상시를 먼저 정하고, 싼 규칙으로 거른 뒤 비싼 AI는 마지막에만.

★

【최우선】 Claude에게 “로그인”을 시키지 마라

제보 · 양현모 · 보안 관련 가장 중요한 항목
⚠ 매우 중요 — 회사 PC면 특히
원래 메모
“Claude Code가 로그인을 하게 되면 SSO나 passkey 같은 인증 토큰을 가져가려 하는데, 이 행동이 근본적으로 해킹과 다르지 않다. 회사 컴퓨터에선 대부분 보안 프로그램에 걸린다. 걸리면 소명하거나 장비를 압수당하는 등 불편해진다. 로그인이 필요하면 Playwright·브라우저·Selenium으로 창은 열되 내가 직접 로그인하고, 그다음 작업을 자동화하라.”
사실확인 & 보충
  • 에이전트가 자동으로 로그인하는 것은 자격증명·세션 토큰을 프로그램이 가로채는 행위입니다. 사내 보안 솔루션(EDR/DLP)은 이를 ‘크리덴셜 탈취’ 패턴으로 탐지합니다 — 의도가 선해도 행위 시그니처가 공격과 동일합니다.
  • SSO·passkey·2FA는 “사람이 직접 인증”하라고 만든 장치입니다. 자동화가 이를 우회·대행하려는 순간 정책 위반이자 탐지 대상이 됩니다.
  • 걸리면: 보안팀 소명, 계정 잠금, 최악의 경우 장비 회수. 시간 낭비에 신뢰까지 잃습니다.
올바른 패턴 — 사람이 로그인, 그다음만 자동화
  • ① Playwright / Selenium / 브라우저로 페이지를 사람이 보는 채로 연다.
  • ② 로그인은 내가 직접 손으로 한다 (ID/PW, 2FA, passkey).
  • ③ 로그인된 세션 위에서 그다음 작업(메일 읽기, 목록 크롤링 등)만 자동화에 넘긴다.

예: 네이버 메일을 읽어와야 하거나 쇼핑 목록을 크롤링해야 하는데 로그인이 필요하면 — 브라우저는 자동화로 띄우되 로그인 단계에서 멈추고 사람이 인증, 그 뒤부터 작업을 이어가게 워크플로우를 짜세요.

✕ 절대 하지 말 것

  • 스크립트에 회사 계정 비번 하드코딩
  • 에이전트에게 로그인 자동 통과 시키기
  • SSO/passkey/2FA 우회 시도

✓ 이렇게

  • 로그인은 항상 사람이 직접
  • 인증 끝난 세션부터 자동화
  • 회사 정책이 의심되면 먼저 확인
핵심

로그인은 무조건 사람 손으로. AI가 인증을 대행하는 순간 보안엔 ‘해킹’으로 보인다.
